Git - 学习笔记

1 Git的出生

在1991年的时候Linux的爸爸Linus创建了开源的Linux系统。从此，Linux在程序界掀起一场腥风血雨，在众多程序猿的拥护中成长为全球最大的服务器系统软件。

那么问题来了，世界各地的人是怎么帮助Linus不断完善Linux的呢？代码又是以何种方式管理的呢？

由于Linux的“开源”精神，开发者社区的兄弟们拒绝以任何付费形式的版本控制系统，但免费控制器的执行速度又太慢，还不如Linus亲自操刀来的快。所以，起先猿友们都把代码发送给主开发者Linus，由Linus通过手工的方式合并代码。

但是，随着Linux的不断成长，代码库也越来越巨大，Linus已经力不从心了，开发者社区的兄弟们也意识到不能再让这种情况发展下去。

于是，在2002年出现了一位金主“BitMover公司”，它是商业版本控制系统BitKeeper的拥有者。
BitMover公司觉得Linux精神可嘉，可以授权让Linux社区的猿友们免费使用他们的版本控制系统。

好景不长，“World peace”的日子很快就过去了。Linux的精神是什么？是开源！社区的那些汉子看见BitKeeper这小娘们天天在眼前晃悠能不动心思？能压抑住内心的悸动？不存在的！那些胆大的糙汉子已经开始试图破解BitKeeper的协议，然而，结局总是不如意的，作案未果反被抓，BitMover公司很生气的撤销了Linux社区的使用权。

事情发展到了这个地步，正常来说，Linus是该去致歉，保证约束手底下的人，杜绝再次发生此类情况。

但是，他是Linus！是Linux的开创者！怎么可能低声下气的活着！道歉，是不可能的。

Linus想，我这都一大帮子人了，还受别人制约像什么样子，是时候改革开放了。Linus花了两周用C语言写了个符合自己人口味的分布式版本管理系统，就是Git！牛逼！

Linux的源码马上就搬到了Git管理！（哈哈，我猜是不是Linus把人家代码给偷了呀🙃）

自此，Git迅速发展成最流行的分布式版本控制系统，Linus怎么就那么棒呢🤪？尤其是2008年，GitHub网站上线，无数的开源项目都迁至GitHub存储，大大的推广了Git。（谁让它们都不收钱呢，哈哈！）

2 Git的结构

在Git中，我们将需要进行版本控制的文件目录称为本地仓库（Repository），这个目录里的所有文件都是通过Git来实现版本的管理，Git会跟踪并记录该目录下发生的所有更新。

Git的区域结构图：

Git的区域流程图：

工作区（WorkSpace）
本地检出，直接操作的文件区域

暂存区（Index）
即索引。保存工作（树）的快照，作为下次提交的内容

存档库（Stash）
临时保存当前的修改

本地仓库（Repository）
.git文件夹保存版本库需要的全部信息（Git版本库的骨架）

远程仓库（Remote）
即远端服务器仓库。可在网络上共享给其他开发者的版本库

3 Git常用术语

术语 定义
仓库(Repository) 包括了所有的版本信息和所有的分支与标记信息
分支(Branches) 每个分支都是独立的、拥有自己完整历史信息的时间线。可以从已有的代码中生成一个新的分支，并且这个分支与其余的分支是完全独立的
标记(Tags) 一个标记记录的是某个分支某个特定时间点的状态。通过标记，可以很方便的切换到标记时的状态
提交(Commit) 提交代码后，仓库会创建一个新的版本。在版本间可以任意切换
修订(Revision) 用来表示代码的一个版本状态。Git通过用SHA1 hash算法表示的id来标识不同的版本。每一个 SHA1 id都是160位长，16进制标识的字符串
